# CONTRIBUTING

## Назначение
Этот файл фиксирует правила внесения изменений в проект Math Brain Trainer.

Даже если проект изначально разрабатывается одним человеком, такие правила помогают:
- не терять структуру;
- поддерживать единый стиль;
- упрощать развитие проекта;
- легче возвращаться к работе после паузы;
- не превращать кодовую базу в хаос.

---

## Главный принцип
Каждое изменение в проекте должно делать одну из двух вещей:
1. улучшать продукт;
2. улучшать структуру проекта.

Если изменение не делает ни того, ни другого, его стоит пересмотреть.

---

## Подход к разработке
Проект развивается пошагово.

Рабочий цикл:
1. Определить один конкретный следующий шаг.
2. Получить от GPT объяснение и полный код файлов.
3. Вручную создать или изменить файлы в VS Code.
4. Проверить, что всё запускается.
5. Зафиксировать изменения.
6. Обновить документацию при необходимости.
7. Только после этого переходить дальше.

---

## Правила изменений

### 1. Один шаг — одна законченная задача
Не нужно в одном изменении одновременно:
- менять архитектуру;
- переделывать UI;
- добавлять новый режим;
- менять хранение данных.

Лучше делать маленькие и понятные шаги.

### 2. Не ломать проект ради улучшений
Если хочется улучшить архитектуру, но это ломает текущую рабочую версию, нужно:
- сначала понять реальную пользу;
- затем запланировать отдельный рефакторинг;
- не смешивать рефакторинг с новой функциональностью без необходимости.

### 3. Сначала чистый MVP
До завершения MVP не добавлять без необходимости:
- backend;
- авторизацию;
- облачную синхронизацию;
- сложный state manager;
- сложные анимации;
- тяжёлые библиотеки.

### 4. UI и логика не смешивать
Нельзя без необходимости держать бизнес-логику внутри компонентов интерфейса.

Предпочтительно:
- UI в components/ и pages/
- доменная логика в features/
- типы в entities/
- сохранение данных в services/
- утилиты в utils/

### 5. Любой новый режим должен быть расширяемым
Если добавляется новый режим тренировки, нужно сразу думать:
- как он описывается типами;
- как он генерирует задачи;
- как он проверяет ответы;
- как его результаты будут сохраняться в статистику.

---

## Работа с GPT
GPT используется как пошаговый технический помощник.

### Правильный формат работы
GPT должен:
- предлагать один логичный шаг;
- объяснять, что именно делается;
- указывать, какие файлы создать или изменить;
- давать полный код файлов;
- объяснять, как проверить результат.

### Неправильный формат работы
Нежелательно:
- просить GPT сразу написать весь проект;
- смешивать setup, архитектуру и сложную логику в одном запросе;
- вставлять в проект код без понимания, что именно он делает;
- принимать большие изменения без запуска и проверки.

---

## Правила по структуре файлов

### Документация
Нужно поддерживать в актуальном состоянии:
- README.md
- CHANGELOG.md
- CURRENT_STAGE.md
- DECISIONS.md
- ROADMAP.md
- TASKS.md

### Исходный код
Рекомендуемая структура:

src/
- app/
- pages/
- components/
- features/
- entities/
- shared/
- services/
- utils/
- data/

### Общий смысл папок
- app/ — корневая логика приложения
- pages/ — экраны
- components/ — переиспользуемые UI-компоненты
- features/ — бизнес-логика по фичам
- entities/ — доменные типы и сущности
- shared/ — общие элементы
- services/ — storage и внешние сервисы
- utils/ — утилиты
- data/ — константы, конфиги, заготовки данных

---

## Правила именования

### Общие правила
- использовать понятные и предсказуемые имена;
- не использовать слишком абстрактные названия;
- не смешивать разные слова для одной и той же сущности.

### Примеры хороших имён
- TrainingMode
- TechniqueType
- MathTask
- SessionResult
- UserStats
- generateTask
- checkAnswer
- saveStats
- loadStats

### Примеры плохих имён
- DataThing
- HelperStuff
- randomUtil
- taskManagerX
- valueHandler

---

## Правила для TypeScript
- использовать явные типы для доменных сущностей;
- не злоупотреблять `any`;
- выносить повторяющиеся типы отдельно;
- следить за тем, чтобы интерфейсы отражали смысл сущности, а не случайный текущий код.

---

## Правила для UI
- интерфейс должен быть простым и чистым;
- элементы управления должны быть крупными и понятными;
- тренажёр должен быть удобен для коротких повторяющихся сессий;
- желательно сразу думать о mobile-friendly layout;
- визуальный шум нужно минимизировать.

---

## Правила для логики тренажёра
Тренажёр — это не просто набор случайных примеров.

Каждый новый режим желательно оценивать по вопросам:
- что именно он тренирует;
- какая в нём техника;
- как пользователь понимает, что прогрессирует;
- можно ли его развивать дальше.

---

## Правила сохранения данных
На этапе MVP используется localStorage.

Важно:
- не обращаться к localStorage хаотично из разных компонентов;
- использовать отдельный слой services/;
- договориться о понятных ключах хранения;
- хранить данные в предсказуемой структуре.

---

## Коммиты
Желательно делать маленькие коммиты.

Примеры хороших коммитов:
- init vite react ts project
- setup tailwind
- add app shell
- add training mode types
- implement addition basics
- add results screen
- save stats to localStorage

Плохие коммиты:
- updates
- changes
- fix
- many improvements

---

## Когда обновлять документацию
Документацию нужно обновлять, если:
- принято новое архитектурное решение;
- завершён этап roadmap;
- изменился текущий этап проекта;
- добавлена важная фича;
- изменился подход к структуре проекта.

---

## Когда нужен рефакторинг
Рефакторинг нужен, если:
- логика начала дублироваться;
- типы стали путаться;
- UI и бизнес-логика слишком смешались;
- добавление нового режима стало слишком трудным;
- код перестал быть очевидным после паузы в работе.

---

## Цель всех правил
Не просто написать приложение, а построить проект, который:
- приятно развивать;
- легко понимать;
- удобно масштабировать;
- не страшно открыть через месяц и продолжить.
